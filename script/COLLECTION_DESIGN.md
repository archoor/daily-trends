# 趋势网站采集方案设计（供选择）

本文档给出几种可选的采集与入库方案，便于你选定方向后再实现。涉及的数据源：**Toolify（AI Tools）**、**GitHub Trending**、**Product Hunt**、**Google 趋势**。

---

## 一、采集范围与数据策略

### 1.1 采集内容

| 数据源 | 列表页 | 详情页 | 说明 |
|--------|--------|--------|------|
| **Toolify** | 必采：排行、工具名、月访问、增长、介绍、标签 | 可选：长描述、原始 JSON | 列表即主数据，详情可后续按需抓取 |
| **GitHub** | 必采：仓库名、描述、语言、星标、Fork、今日新增、链接 | 可选：README 摘要等 | 列表通常一页即可，可按 today/weekly/monthly 各采一份 |
| **Product Hunt** | 必采：排行、产品名、描述、图标、类别、评论数、点赞数、链接 | 可选：详情描述 | 可能有分页或“加载更多” |
| **Google 趋势** | 必采：趋势名、搜索量、增长率、开始/结束时间、相关词 | 可选：详情 | 需确认目标页面（如每日/实时趋势列表） |

**建议**：先做**列表页每日全量**，详情页可做成“按访问触发”或单独的低频任务。

### 1.2 数据更新策略（选一种）

- **方案 A：每日全量快照**  
  每天固定时间抓取各站当前列表，以 `snapshotAt = 当天 0 点` 写入，历史快照保留。  
  - 优点：实现简单、可做历史对比；  
  - 缺点：存储会随天数增长，需定期归档或清理旧快照。

- **方案 B：每日覆盖**  
  每天只保留“最新一天”的数据：先删该数据源当天或昨天的记录，再写入今日列表。  
  - 优点：库表体积小、逻辑简单；  
  - 缺点：无历史趋势对比。

- **方案 C：每日追加 + 定期清理**  
  每日追加新快照，保留最近 N 天（如 30 天），超过 N 天的快照由定时任务删除。  
  - 优点：兼顾历史与存储；  
  - 缺点：需多一个清理脚本或逻辑。

**推荐**：若要做“趋势变化”分析选 **A 或 C**；若只展示“今日榜单”选 **B**。

## 二、脚本组织方式

### 2.1 方案一：按数据源分脚本（推荐起步）

```
script/
├── README.md
├── requirements.txt
├── _common/                 # 可选：公共配置、DB 连接、日志
│   ├── db.py
│   └── config.py
├── collect_toolify.py       # 只采 Toolify
├── collect_github.py        # 只采 GitHub
├── collect_producthunt.py   # 只采 Product Hunt
└── collect_google.py        # 只采 Google
```

- **运行方式**：手动或 cron 分别执行 4 个脚本，例如每天 6:00 依次跑。
- **优点**：边界清晰、某个站出错不影响其他站；便于单独调试、单独重跑。
- **缺点**：调度要自己用 cron 或任务系统串起来。

### 2.2 方案二：统一入口 + 按源分模块

```
script/
├── _common/
│   ├── db.py
│   └── config.py
├── collectors/
│   ├── __init__.py
│   ├── toolify.py
│   ├── github.py
│   ├── producthunt.py
│   └── google.py
├── run_all.py               # 入口：可指定 --sources toolify,github 或全部
└── requirements.txt
```

- **运行方式**：`python run_all.py` 或 `python run_all.py --sources github,toolify`。
- **优点**：一次调用跑多源；可统一日志、统一重试与告警。
- **缺点**：一个源报错需要你决定是“继续其他源”还是“整体失败”。

### 2.3 方案三：队列 + 独立 Worker（适合多机或复杂调度）

- 用一个任务队列（如 Celery + Redis，或 SQLite 做简单队列），每天由 cron 往队列里投递 4 个“采集任务”。
- 一个或多个 Worker 消费任务，每个任务只跑一个数据源。
- **优点**：易扩展、可重试、可多机；  
- **缺点**：依赖与运维更重，适合后续规模上来后再上。

**建议**：当前阶段用 **方案一** 或 **方案二** 即可。

## 三、每日调度方式

### 3.1 本机 Cron（最直接）

- 在服务器或本机用 crontab 每天固定时间执行，例如 6:00：
  - 若用方案一：`0 6 * * * cd /path/to/script && python collect_toolify.py`，同理 4 条。
  - 若用方案二：`0 6 * * * cd /path/to/script && python run_all.py`。
- **优点**：零额外依赖，易理解。  
- **注意**：需保证机器该时段在线，且环境（Python、依赖、DATABASE_URL）正确。

### 3.2 Windows 任务计划程序

- 在 Windows 上可用“任务计划程序”新建每日 6:00 运行的任务，操作对象同上（方案一或方案二的命令）。
- **优点**：不依赖 Linux，适合你当前在 Windows 开发/跑脚本。

### 3.3 云函数/Serverless（如腾讯云 SCF、阿里云 FC）

- 为每个数据源一个定时触发器（或一个函数内按源顺序调用），每天触发一次。
- **优点**：不用常开机器；  
- **缺点**：需把 DB 放到云上或暴露（不推荐公网直连），或通过 API 写入你本地的服务，实现复杂度较高。

### 3.4 容器/轻量 CI（如 GitHub Actions）

- 在仓库里放一个 workflow，每天 6:00 跑一次“采集 job”：拉代码、装依赖、执行 `run_all.py`。
- 数据库需对 Actions 可达（例如临时用云上 DB，或自建带公网 + 鉴权）。
- **优点**：不占本机、可复用仓库权限与日志；  
- **缺点**：需要可被访问的数据库和一定的网络配置。

**建议**：先在本机用 **Cron** 或 **Windows 任务计划程序** 跑通，再考虑云或 CI。

## 四、技术实现要点（不写代码，只定策略）

### 4.1 抓取方式

- **Toolify / Product Hunt**：多为前端渲染，列表很可能在 XHR/Fetch 里返回 JSON，优先用**浏览器开发者工具看接口**，能调 API 就直接请求；否则用 **Playwright** 或 **Selenium** 渲染后取数。
- **GitHub Trending**：页面是服务端渲染，可用 **requests + BeautifulSoup** 直接解析 HTML；若遇反爬再考虑 Playwright。
- **Google 趋势**：通常需 **Playwright**（或带 Cookie/登录的请求），且要注意 Google 的反爬与频率限制。

### 4.2 与数据库的配合

- 使用项目根目录 `.env` 的 `DATABASE_URL`（或 script 下同名的 `.env`），用 **SQLite** 时注意并发：同一时间尽量只有一个写进程。
- 写入前先确保 **DataSource** 存在（如 `slug in ('toolify','github','producthunt','google')`），没有则插入。
- 利用 schema 的 **@@unique([sourceId, externalId, snapshotAt])**：同一源、同一条目、同一天只保留一条，重复写入用 upsert（INSERT OR REPLACE / ON CONFLICT DO UPDATE）即可实现幂等。

### 4.3 礼貌爬取与稳健性

- 每个请求之间 **加随机延时**（如 1～3 秒），避免短时间大量请求。
- **User-Agent** 设为常见浏览器，必要时带上 Referer。
- 单源失败：记录日志并跳过该源，不影响其他源；下次调度再重试。
- 可设**超时**（如单源整体 5 分钟），超时则放弃当次该源，避免卡死。

### 4.4 日志与可观测

- 日志建议包含：开始/结束时间、数据源、抓取条数、写入条数、错误信息。
- 可简单写到一个 `script/logs/` 目录，按日期或按天轮转，便于排查“某天某源没数据”的问题。

## 五、方案组合推荐（供直接选择）

| 组合 | 数据策略 | 脚本组织 | 调度方式 | 适用场景 |
|------|----------|----------|----------|----------|
| **1** | B：每日覆盖 | 方案一：4 个独立脚本 | Windows 任务计划 / Cron | 只展示“今日榜”，本机跑，实现最快 |
| **2** | A 或 C：保留快照 | 方案二：run_all + collectors | Cron 每日一次 | 要历史对比，单机即可 |
| **3** | C：追加 + 定期清理 | 方案二 + 清理脚本 | Cron（采集 + 每周清理） | 平衡存储与历史，稍复杂 |
| **4** | A：全量快照 | 方案三：队列 + Worker | 云或自建服务器 Cron | 多源、多机、后续要扩展时 |

## 六、需要你拍板的点

1. **历史数据**：要保留多天做对比（A/C），还是只保留“最新一天”（B）？  
2. **脚本结构**：先 4 个独立脚本（方案一）还是统一入口 run_all（方案二）？  
3. **调度环境**：目前是 Windows 本机、Linux 服务器，还是打算用云/CI？  
4. **详情页**：是否第一版就做详情页采集，还是列表跑稳后再加？

你选定上述选项后，可以再针对“方案组合”和“数据源优先级”（例如先做 GitHub + Toolify）细化到具体脚本接口与表操作步骤，然后再进入编程实现阶段。
